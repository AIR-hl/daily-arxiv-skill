# 飞书渲染

这个文件只在论文精读报告需要保存到飞书文档时读取。

## 三阶段流程

### Compose

- 先根据 `assets/templates/paper_template.md` 组织论文精读内容，保持模板定义的章节顺序和写作责任。
- Compose 阶段的目标是内容正确、论证完整、结构清晰；它不是最终飞书阅读形态。
- 不要因为后续要做飞书渲染而改变模板的核心章节，也不要把报告写成 checklist。

### Render

- 在执行飞书文档创建或更新前，把 Compose 后的内容转换为适合飞书阅读的发布稿。
- Render 阶段采用自适应策略：根据论文内容选择能提高阅读效率的飞书呈现方式，不固定每个章节必须使用的组件，也不要为了使用组件而破坏文章节奏。
- 如果使用飞书扩展能力，请参考相关 SKILL 说明，不要在凭记忆硬写具体语法。

### Publish

- 发布到飞书时，传给飞书文档创建或更新操作的应是面向飞书阅读体验处理过的内容，不要把未经任何阅读体验审视的模板化正文直接发布。

## 渲染目标

- 让读者先看到论文的核心判断，再进入论证细节。例如：开头可以用提示块概括本文主张、证据强度和最终判断，而不是先铺长段背景。
- 让风险、局限、证据强弱和审稿判断在视觉上容易扫到。例如：对“实验证据不足”“只在窄场景成立”“baseline 不公平”等判断，可以用醒目标注、颜色或高亮处理。
- 让关键定义、作者原始立场、重要假设和方法对比更容易定位。例如：关键概念可以放入引用样式或知识卡片；多个方法、实验设置、消融结果或风险点可以改成表格。
- 让机制流程、模块关系、因果链、训练/推理路径尽可能图文化或结构化。例如：如果论文原图对理解有帮助，可以插入原图；如果需要重新解释流程，可以使用画板、流程图、ASCII 图或分步骤结构。
- **保持克制**：组件应该服务于理解，不要为了“看起来丰富”而给每段都加各种组件渲染。

## 发布前检查

- 检查最终发布稿是否仍然保留了 `paper_template.md` 的内容责任：动机、彻底剖析、核心概念、关键细节、博导审稿、总结与洞见。
- 检查最终发布稿是否经过飞书阅读场景下的可读性审视，包括层次、重点、图示、对比和扫读路径。
- 如果最终发布稿接近纯文本，也可以发布，但要先确认这是出于内容需要，而不是遗漏了 Render 阶段。
